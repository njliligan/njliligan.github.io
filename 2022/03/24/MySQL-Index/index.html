MySQL优化

Created2022-03-24|Updated2022-03-29

概述

数据库性能一般取决于几个因素，比如表，查询，配置，这些软件结构导致硬件级别的CPU和I/O，必须尽可能使其最小化来提高效率，

1，在数据库的级别进行优化

使数据库快速运行的最重要的因素是他的基础设计，
1，表结构是否正确，特别是列是否有正确的数据类型，每个表都有合适工作类型的列，例如，频繁更新的程序有很多的表和很少的列，大量数据的则有很少的表和大量的列。
2，是否有正确的索引来提高查询效率。
3，对于每个表是否使用适当的储存引擎，并且使用了特性来提高效率。
4，每个表是否使用适当的行格式，这个还取决于表的储存引擎，特别是压缩表使用更少的磁盘空间，所以可以用更少的磁盘I/O来读取和写入数据，压缩适用于具有 InnoDB 表的所有类型的工作负载，以及只读 MyISAM 表。
5，应用程式是否使用的恰当的锁，例如，尽可能允许共享访问，以便数据库操作可以并发运行，并在适当时请求独占访问，以便关键操作获得最高优先级。再次，存储引擎的选择意义重大，InnoDB 存储引擎可以在你不参与的情况下处理大多数锁定问题，从而在数据库中实现更好的并发性并减少代码的实验和调整量。
6，用于缓存的所有内存区域的大小是否正确，打到足以容纳经常访问数据，但是不会大到使物理内存过载分页，最主要是配置InnoDB的缓冲池和MyISAM的key缓存。

2，在硬件层面的优化

随着数据库越来越繁忙，任何数据库最后都会受到硬件的限制，一个DBA必须评估是否可以调整应用程序或重新配置服务器以避免这些瓶颈，或者分配更多的硬件资源，系统瓶颈通常来源于这些。
1，磁盘寻道，磁盘找到一条数据需要的时间，这个时间随着新磁盘的使用缓慢增加，并且很难针对单个表进行优化。优化寻道时间的方法是将数据分布到多个磁盘上。
2，磁盘读写，当磁盘在正确的位置时，我们需要读取或写入数据。使用现代磁盘，一个磁盘可提供至少 10–20MB/s 的吞吐量。这比查找更容易优化，因为您可以从多个磁盘并行读取。
3，CPU周期，当数据在主要内存中，我们必须处理它以获得结果，与内存量相比，大表是更常见的限制，对于小表来说，速度通常不是问题，
4，内存带宽，当 CPU 需要的数据超出 CPU 高速缓存的容量时，主内存带宽就会成为瓶颈。对于大多数系统来说，这个不常见，但需要注意。

3，平衡可移植性和内存

要在可移植的 MySQL 程序中使用面向性能的 SQL 扩展，您可以将 MySQL 特定的关键字包装在 /*! */的注释分隔符语句中。其他 SQL 服务器忽略注释的关键字。

多值比较的等式范围优化(Equality Range Optimization of Many-Valued Comparisons)
col_name是一个索引行

如果 col_name 等于多个值中的任何一个，则每个表达式都为真。这些比较是相等范围比较（其中“范围”是单个值）。优化器估计读取符合条件的行以进行相等范围比较的成本如下：
如果 col_name 上存在唯一索引，则每个范围的行估计值为 1，因为最多一行可以具有给定值
否则， col_name 上的任何索引都是不唯一的，优化器可以使用深入索引或索引统计信息来估计每个范围的行数。
使用索引dives，优化器在范围的每一端进行dives，并使用范围内的行数作为估计值，例如，表达式 col_name IN (10, 20, 30) 具有三个相等范围，优化器对每个范围进行两次dives以生成行估计。每对dives都会产生具有给定值的行数的估计值。
(之所以出现这个，是因为有两种估算方式，一是dive到index中即利用索引完成元组数的估算,简称index dives; 二是使用索引的统计数值,进行估算.)
索引dives提供准确的行估计，但是随着表达式中比较值数量的增加，优化器需要更长的时间来生成行估计。索引统计的使用不如索引dives准确，但允许对大值列表进行更快的行估计。
eq_range_index_dive_limit 系统变量使您能够配置优化器从一种行估计策略切换到另一种，要允许使用索引dives来比较最多 N 个相等范围，请将 eq_range_index_dive_limit 设置为 N + 1，要禁用统计信息并始终使用索引dives而不考虑 N，请将eq_range_index_dive_limit 设置为 0
要更新表索引统计信息以获得最佳估计，请使用 ANALYZE TABLE
在 MySQL 8.0 之前，除了使用 eq_range_index_dive_limit 系统变量外，没有办法跳过使用索引dives来估计索引有用性，在 MySQL 8.0 中，满足所有这些条件的查询可以跳过索引dives：
1，查询是针对单表的。
2，存在单索引 FORCE INDEX 索引提示。这个思想是，如果强制使用索引，那么执行深入索引的额外开销将没有任何好处。
3，该索引是非唯一的，不是 FULLTEXT 索引。
4，不存在子查询。
5，不存在 DISTINCT、GROUP BY 或 ORDER BY 子句。
对于 EXPLAIN FOR CONNECTION，如果跳过索引dives，输出将更改如下
1，对于传统输出，行和过滤值是 NULL。
2，对于 JSON 输出，rows_examined_per_scan 和 rows_produced_per_join 没有出现，skip_index_dive_due_to_force 为真，成本计算不准确。
如果没有 FOR CONNECTION，则跳过索引dives时 EXPLAIN 输出不会改变。
在执行跳过索引dives的查询后，INFORMATION_SCHEMA.OPTIMIZER_TRACE 表中的相应行包含 skipped_due_to_force_index 的 index_dives_for_range_access 值。

4，索引前缀(Index Prefixes)。

Prefixes can be up to 767 bytes long for InnoDB tables that use the REDUNDANT or COMPACT row format. The prefix length limit is 3072 bytes for InnoDB tables that use the DYNAMIC or COMPRESSED row format. For MyISAM tables, the prefix length limit is 1000 bytes.

前缀是按照字节为单位的，所以当你使用字符类型的列要特别注意。

1，包含引用不可见索引的索引提示的查询会发生错误。
2，性能消耗随受影响的查询增加。
3，查询有不同的执行计划。
4，查询出现在之前没有出现在慢查询日志中。

9，在索引列上使用IS NULL或者IS NOT NULL，看情况失效。

10，NOT IN，NOT EXIST，索引会失效。

11，在索引字段上使用<>，!=。（整形索引中使用<,>不会使索引失效。主键索引使用!=,>,<都不会失效）不等于操作符是永远不会用到索引的，因此对它的处理只会产生全表扫描。 优化方法： key<>0 改为 key>0 or key<0。
