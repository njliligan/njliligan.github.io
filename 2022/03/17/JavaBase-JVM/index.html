调整参数记忆方法

参数分类

根据jvm参数开头类型一共三类 “-“，”-X”，”-XX”,
“-“：表示标准参数，所有的JVM都必须实现这些参数的功能，并且向后兼容
例如：-verbose:gc
“-X”：表示非标准参数，默认jvm实现这些参数的功能，但是不保证所有的jvm实现都能满足，且不保证向后兼容:
例如：-Xms20m，
“-XX”：非Stable参数：此类参数各个jvm实现会有所不同，可能随时取消
例如：-XX:+PrintGCDetails

含义拆分

例如：-Xms20m：

参数 拆分 含义
-Xms -X,memory,size 内存大小
-Xmx -X,memory,max 内存最大值
-Xmn -X,memory,new 新生代内存
-Xss -X,stack,size 栈大小

类加载

1,加载

作用：
将Class文件以二进制流的方式加载到java虚拟机并储存到方法区中
流程：
1，通过类的全名限定来获取二进制字节流
2，将这个字节流所代表的静态存储结构转化为方法区的运行时数据结构
3，在内存中生成一个代表这个类的java.lang.Class对象，最为方法区该类的各种数据的访问入口
注意：
数组类通过jvm直接创建不通过类加载器

2，验证

作用：
确保加载进来的Class文件包含的星系符合jvm的要求
流程：
1)，文件格式验证
2)，元数据验证
3)，字节码验证
4)，符号引用验证

3，准备

作用：
为类变量分配内存，设置类变量的初始值
流程：
1，为类的static变量在方法区中分配内存
2，将变量的初始值设置为”0”,根据类型变化
注意：
实例变量不在这个阶段分配内存
如果类变量给常量(被final修饰)，值等于开发者定义的值

4，解析

作用：
将常量池内的符号引用变为直接引用
流程：
解析对象包括
1，类/接口
2，方法
​ 1)类方法
​ 2)接口方法(私有)
​ 3)方法类型
​ 4)方法句柄
3，字段
注意：
实例变量不在这个阶段分配内存
因为类方法和私有方法符合“编译器可知，运行期不可变”的要求，不会被继承或者重写，所以比较适合在类加载的过程中进行解析
若类变量为常量（被final修饰），值等于开发者定义的值

5，初始化

作用：
初始化类变量和静态代码块
具体流程：
1，生成类构造器方法<clinit>方法
​ 1)合并所有的类变量（static变量）和静态代码块（static块）
​ 2)顺序 = 定义的顺序，实例变量由类的构造器<init>()赋值
2，执行<clinit>()方法
注意：
类构造器<clinit>()区别于类构造函数<init>()
​ 1，不需要显示调用父类构造器
​ 2，子类<clinit>()执行前，父类的<clinit>()一定会被执行
​ 3，所以虚拟机第一个执行的<Clint>()是java.lang.Object
静态语句块只可赋值，不可被访问
接口与类不同执行子接口的<clinit>()并不需要执行父接口的<clinit>()
init是instance实例构造器，对非静态变量解析初始化，而clinit是class类构造器对静态变量，静态代码块进行初始化

Heap

1,介绍

OOM发生的主要区域，内存中最大的一块区域，被所有线程共享，储存着几乎所有的实例对象和数组(不绝对)。是垃圾收集器管理的主要区域,基本采用分代收集算法，堆中还可以细分为：新生代和老年代。再细致一点的有 Eden 空间、From Survivor 空间、To Survivor 空间，从内存分配的角度来看，线程共享的 Java 堆中可能划分出多个线程私有的分配缓冲区（Thread Local Allocation Buffer,TLAB）。

对象创建内存分配流程

绝大部分对象在 Eden 区生成，当 Eden 区装填满的时候就会触发gc，在 Eden 区实现的清除策略是，没有被引用的对象则直接回收。依然存活的对象会被移送到 Survivor 区。Survivor 区分为 so 和 s1 两块内存空间。每次 gc 的时候，它们将存活的对象复制到未使用的那块空间，然后将当前正在使用的空间完全清除，交换两块空间的使用状态。就是s0和s1两个空间，一次只有一个空间在被使用，如果 gc要移送的对象大于 Survivor 区容量的上限，则直接移交给老年代。

－XX:MaxTenuringThreshold 参数就是来配置一个对象从新生代晋升到老年代的阈值。默认值是 15，可以在 Survivor 区交换 14 次之后，晋升至老年代。

元空间

1，在原来永久代中存放了类和方法的元数据以及常量池，例如Class和Method。当每一个类被加载的时候，元数据都是放到永久代。但是由于永久代容易发生OOM，同时促进HotSpotJVM和JRockit VM的融合，取消了永久代。将类元信息、字段、静态属性、方法、常量等都移动到元空间(MetaSpace)，字符串常量池移到了堆当中。空间并不在虚拟机中，而是使用本地内存。

没有压缩的类空间

启用压缩类空间后，我们有两个元空间上下文（一个正常，一个包装类空间），每个 CLD 现在有两个领域，一个与非类上下文相关联，一个与类空间相关联。 在这种模式下，Klass 结构的内存从类空间分配，所有其他元空间分配从非类元空间：

JVM Stacks 中stack的结构

对于每一个线程，JVM 都会在线程被创建的时候，创建一个单独的栈。也就是说虚拟机栈的生命周期和线程是一致，并且是线程私有的。除了Native 方法以外，Java 方法都是通过 Java 虚拟机栈来实现调用和执行过程的（需要程序技术器、堆、元空间内数据的配合）。所以 Java 虚拟机栈是虚拟机执行引擎的核心之一。而 Java 虚拟机栈中出栈入栈的元素就称为「栈帧」。一个栈有多个栈帧，每个栈帧结构如下，另外由于栈的特性，先进后出，所以在，在活动线程中， 只有位于栈顶的帧才是有效的， 称为当前栈帧。正在执行的方法称为当前方法。在执行引擎运行时， 所有指令都只能针对当前栈帧进行操作。而 StackOverflowError 表示请求的栈溢出， 导致内存耗尽， 通常出现在递归方法中。虚拟机找通过pop和push的方式，对每个方法对应活动栈进行运行算，真唱结束后会跳转到另一个栈帧，出现了异常就会发生异常回溯，返回地址通过异常处理表确定。

1，局部表量表

部变量表就是存放方法参数和方法内部定义的局部变量的区域。

2，操作栈

操作数栈（Operand Stack）看名字可以知道是一个栈结构。Java 虚拟机的解释执行引擎称为“基于栈的执行引擎”，其中所指的“栈”就是操作数栈。当 JVM 为方法创建栈帧的时候，在栈帧中为方法创建一个操作数栈，保证方法内指令可以完成工作。

3，动态链接每个栈帧包含一个在常量池中对当前方法的应用，目的是支持方法调用过程的动态链接

4，返回地址

方法执行有两种退出的情况
1，正常退出。
2，异常退出。
无论何种情况，都将返回方法当前被调用的位置，退出过程相当于弹出了当前帧，可能有三种方式
1，返回值压入上层调用栈帧。
2，异常信息抛给能处理的栈帧。
3，PC计数器指向方法调用后的下一条指令

本地方法栈

本地方法栈为虚拟机使用到的 Native 方法服务。其他与JVM栈相似

程序计数器

程序计数器（Program Counter Register）是一块较小的内存空间。是线程私有的。它可以看作是当前线程所执行的字节码的行号指示器。在程序的处理中，线程可能会在执行过程中被挂起，而程序在被挂起，然后继续执行的时候，CPU需要知道执行程序的那一部分的指令。线程计数器就是起到这个作用，CPU只有把数据装载到寄存器才能运行，寄存器储存指令相关的现场信息，各个线程之间计数器互不影响，独立储存，程序计数器用来存放执行指令的偏移量和行号指示器等，线程的执行和回复都需要程序计数器，不会发生内存溢出异常。

直接内存

主要是由于NIO类的Channel和BUffer的出现，它可以使用 Native 函数库直接分配堆外内存，然后通过一个存储在 Java 堆中的 DirectByteBuffer 对象作为这块内存的引用进行操作。这样能在一些场景中显著提高性能，因为避免了在 Java 堆和 Native 堆中来回复制数据。 也会出现OOM，

Code Cache

JVM 将其字节码存储为本机代码的区域 ，也可能会OOM。

补充

1，内存中的基地址和偏移量(偏移地址)

CPU和内存之间通过地址总线连接，地址总线就是cpu通过地址找到对应的内存的物理数据的传递工具，如果一个地址表示一个存储单元，那么假如说我有20条总线，一共就能存储2^20个地址的数据，大约1M的数据(现在的AMD大概有52位物理总线了)。最开始总线不足的情况，人们提出可以将地址分段，比如，将1m的空间分为16块，每64k分为一个段，将原有的直接表示内存的方式改为：”段地址:段内地址” 的方式，转换下概念实际上就是 “基地址:偏移地址”，当基地址每次加一，在外部的内存地址就偏移了16，计算：段地址x16+基地址 = 实际物理内存的地址 。
